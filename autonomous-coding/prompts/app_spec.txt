Build a full-stack AI chat application with the following goals:

- A ChatGPT or Codex style chat interface for one-on-one and multi-turn conversation
- A modern React and Vite frontend
- A Node and Express backend
- SQLite for local persistence
- OpenAI Responses API for model interaction
- SSE streaming for incremental assistant output
- Conversation management with create, rename, archive, and resume flows
- Artifact handling for files and generated outputs
- Model selection and user settings
- Usage tracking and lightweight history views
- Accessibility, keyboard support, and responsive design

Product requirements:

1. The UI should feel like a practical coding assistant, not a generic marketing site.
2. The chat surface should support long responses, code blocks, message metadata, and clear loading states.
3. Conversation state should persist across reloads.
4. Streaming should begin quickly and remain stable during long generations.
5. The interface should work on narrow and wide viewports.
6. The design should support keyboard navigation, readable contrast, and semantic structure.
7. Settings should include model choice, conversation preferences, and display options.
8. Usage data should be visible enough to help a user understand recent activity.
9. Artifacts should be attached to a conversation and easy to reopen.
10. The implementation should be practical for local development and testing.

Suggested architecture:

- Frontend: React, Vite, accessible component structure, responsive layout
- Backend: Node and Express REST endpoints plus SSE stream endpoints
- Database: SQLite with tables for conversations, messages, settings, artifacts, and usage
- API layer: OpenAI Responses API client with streaming support

Non-goals:

- Do not depend on provider-specific services outside the OpenAI stack.
- Do not require a browser extension or hosted external state to function.
- Do not make the experience depend on animations or visual effects for core usability.

Acceptance criteria:

- A user can create a conversation, send a message, receive a streamed response, reload the page, and continue the same conversation.
- A user can change models and settings without breaking existing history.
- A user can review past conversations and attached artifacts.
- The interface remains usable when the viewport is small.
- The app can be run locally with standard install and start commands.
